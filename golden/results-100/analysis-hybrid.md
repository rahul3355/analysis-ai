## Mechanism: How Classification Errors Cascade

The Pinecone intent cache at `apps/web/src/server/services/intentCacheService.ts:19-43` stores the result of `classifyIntentFull` for each question. When the same question is asked again, the cached intent is returned with `stage: "cache"` and `confidence: 0.95`, bypassing both heuristic and LLM classifiers entirely.

The cached intents for 11 of 16 failed questions are **wrong** (stored as `DOCUMENT` or `DATABASE` instead of `HYBRID`). Once cached, the orchestrator only runs one pipeline branch:

```
Cached as DOCUMENT → only execDoc() runs → BQ never queried
Cached as DATABASE → only execBq() runs → RAG never queried
Cached as HYBRID   → both run (P2/P4 failures only)
```

The orchestrator at `apps/web/src/core/pipeline/orchestrator.ts:77-115` shows this branching:

```typescript
// Line 77-94: DATABASE branch — only execBq()
// Line 95-110: DOCUMENT branch — only execDoc()
// Line 111-115: HYBRID branch — both execDoc() and execBq()
```

When classified as `DOCUMENT`, the document context is sent to the LLM without any BQ data. The LLM answers from documents alone, producing numbers like "46.8%" (plan figure), "51.5%" (Q3 doc figure), or "113,000 units" (wrong document number).

When classified as `DATABASE`, only BQ data is sent but document context is missing. The LLM misses the target/plan figures from documents.

The cached intent is DATABASE, so only `execBq()` ran. The SQL used `product_name = 'Clifton 9'` but the actual name is `'Hoka Clifton 9'`. The product_name includes the brand prefix, similar to hyb-04. BQ returned 0 rows.

**Summary:** Same product_name SQL bug as hyb-04. The NL-to-SQL model strips the brand prefix from product names.

## Failure Pattern Breakdown

### Pattern 1 — Classification Error (11/16 = 69%)

**Mechanism:** Pinecone intent cache stores wrong intent (DOCUMENT or DATABASE instead of HYBRID). Once cached, `classifyIntentFull` returns the cached intent with `stage: "cache"` and `confidence: 0.95` (`intentCacheService.ts:19-43`), completely bypassing the heuristic and LLM classifiers. The orchestrator then only runs the corresponding pipeline branch, missing either BQ data or document context.

**Root cause of cache poisoning:** The initial classification (when the cache was written) likely came from:
1. **Heuristic classifier** matching strong DOCUMENT patterns (e.g., `/gross margin/i` at 0.95) or DATABASE patterns (e.g., `/bigquery/i` at 0.92) without recognizing the HYBRID nature
2. **LLM classifier** (OpenRouter `deepseek/deepseek-v4-flash`) classifying hybrid questions as single-intent

The heuristic classifier at `classifierHeuristics.ts:113-115` checks HYBRID patterns first:
```typescript
if (scores.HYBRID > 0) {
  return { intent: "HYBRID", confidence: scores.HYBRID, ... };
}
```
But many questions don't trigger HYBRID patterns because:
- The patterns are too narrow (`/(actual|real).{0,10}(vs|versus|vs\.).{0,10}(plan|target|budget)/i` requires proximity)
- The "conflicting-patterns" resolution at lines 129-133 correctly identifies HYBRID only when two patterns from different categories both score >= 0.7

Once the wrong intent is cached in Pinecone (namespace `intent-routing-cache`), every subsequent run gets the wrong answer.

**Fix:** Add TTL to Pinecone cache entries, or validate cached intents against heuristic before returning.

### Pattern 2 — SQL Generation Error (3/16 = 19%)

**Mechanism:** The NL-to-SQL model (`cohere/north-mini-code:free`) generates syntactically valid SQL that returns wrong or empty results:

1. **Wrong product names** (hyb-04, hyb-23): The model strips brand prefixes. `product_name = 'Air Force 1 Low'` instead of `'Nike Air Force 1 Low'`. This is the most common SQL bug.

2. **Missing JOIN logic** (hyb-15): Complex 4-table JOIN for returns-by-region likely produced wrong results.

3. **SUM of empty set treated as "0 records"** (hyb-04): The orchestrator's `hasActualData` check at line 145-147 treats a row of all-NULL values as "no data." This causes the system to report "0 records found" when actually the SQL was wrong.

### Pattern 4 — Context Merging Failure (2/16 = 12%)

**Mechanism:** Both RAG and BQ execute successfully, but the LLM fails to use BQ data:

1. **Document context dominates** (hyb-02, hyb-21): The system prompt at line 169 concatenates document chunks and BQ results with `parts.join("\n\n")`. When document context is rich with relatable numbers, the LLM defaults to those numbers instead of deriving answers from BQ raw data.

2. **BQ data is not pre-computed** (hyb-02): The `summarizeRow` function (line 22-24) truncates to 4 columns. The LLM must compute gross margin from raw revenue/cost — but it has access to the pre-computed percentage from the document and defaults to it.

3. **"BQ" literal interpretation** (hyb-21): The LLM treats the string "in BQ" as a literal label instead of recognizing it means "in BigQuery."

## Code Path Analysis

### The Pinecone Cache Poisoning Loop

```
Initial run (no cache):
  1. classifyIntentFull(message)
  2. Heuristic returns INTENT (maybe wrong)
  3. confidence >= 0.85? → cache INTENT in Pinecone (line 53 or 116 of classifier.ts)
  4. Return INTENT to orchestrator

Subsequent runs (cache hit):
  1. classifyIntentFull(message)
  2. getCachedPineconeIntent(embedding) → returns INTENT (wrong!)
  3. Return without running heuristic or LLM classifier
  4. Orchestrator executes only the cached-intent branch
```

The cache write at line 53 (heuristic path) and line 116 (LLM path) does not validate whether the HYBRID intent is appropriate. It unconditionally caches whatever `classifyByHeuristics` or the LLM returns.

### The Product Name SQL Bug

The SQL generator prompt at `bigquerySqlGenerator.ts:72` includes:
```
AVAILABLE TABLES (actual columns from BigQuery):
jd_sports.products:
  - product_name (STRING): Product display name
```

The NL-to-SQL model sees `product_name` as "Product display name" but has no examples of full product names including brand prefixes like "Nike Air Force 1 Low". The golden examples or similar queries in the prompt may use bare names, leading the model to drop the brand prefix.

### The "Empty Result" Ambiguity

At `orchestrator.ts:145-153`:
```typescript
const hasActualData = rows.length > 0 && !rows.every((row) =>
  Object.values(row).every((val) => val === null || val === undefined || val === "")
);
```

A SQL query that runs successfully but returns NULL values (due to wrong WHERE clause) is indistinguishable from a query that genuinely has no matching data. The system says "0 records found" in both cases, but the root causes are different. The LLM takes this literally and reports no data, even when the correct SQL would return useful results.

## Recommendations

1. **Add TTL to Pinecone cache entries** in `intentCacheService.ts:68-71` to allow re-classification
2. **Validate cached intents** by running heuristic check before returning from cache; if heuristic disagrees, re-run LLM classifier
3. **Fix product name matching** in `bigquerySqlGenerator.ts:52-75` by adding instructions to use `LIKE '%name%'` or include the brand prefix
4. **Improve empty-result detection** in `orchestrator.ts:145-153` to distinguish NULL-in-row from no-rows
5. **Add clarifying prompt** in `orchestrator.ts:169` telling the LLM that "BQ" means "BigQuery" and should not be taken literally
6. **Run verify.sh** after fixes to ensure no regressions
